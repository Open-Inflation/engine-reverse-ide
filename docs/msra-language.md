# MSRA Language Reference

MSRA - это декларативный язык описания API-контракта, браузерных сценариев, переменных, регулярных правил, групп, функций и примеров вызова.
Он описывает не только URL и параметры, но и то, как клиент должен быть сгенерирован, как валидировать входы и как строить runtime-поведение.

Эта страница - полный справочник по языку: что делает каждая настройка, зачем она нужна и как она используется в проекте.
Если нужна именно семантика путей и quoted/unquoted-имен, см. отдельную страницу [Пути и identity](msra-paths.md).
Если нужен список того, что текущий codegen ещё не умеет, см. [Ограничения генератора](msra-generator-gaps.md).

## Базовый синтаксис

MSRA-файл состоит из таблиц и assignment-ов.

```msra
[app]
name="MyAPI"

[app.func.GET_USER]
name="get_user"
method=GET
```

### Основные формы значений

| Форма | Что это | Когда использовать |
| --- | --- | --- |
| `строка` | Обычное строковое значение | Для имен, описаний, URL-частей, сообщений об ошибках |
| `123`, `3.14` | Число | Для `timeout_ms`, диапазонов, идентификаторов |
| `true`, `false`, `null` | Булево или null | Для флагов и опциональных значений |
| `[...]` | Массив | Для списков, `authors`, `examples`, `pipeline`, `types`, `values` |
| `{...}` | Inline table | Для вложенных конфигураций, `body`, `headers`, `list_style`, `revalue` |
| `<ROOT.path>` | Ссылка | Для подстановок и связей между разделами языка |

### Кратко о ссылках

Ссылки используются для:

- подстановки префиксов и regex-правил;
- связи `input` и `url.params`;
- чтения `VARIABLES`, `INPUT`, `UNSTANDARD_HEADERS`, `CAPTURED_URLS`, `COOKIES`, `LOCAL_STORAGE`, `SESSION_STORAGE`;
- связывания примеров через `FUNCRESULT`.

Подробнее о синтаксисе ссылок и правилах quoted/unquoted-имен см. [Пути и identity](msra-paths.md).

## Спецкорни и ссылки

| Корень | Что означает |
| --- | --- |
| `DOCUMENT.PREFIXES.*` | Префиксы, объявленные в `[app.prefixes]` |
| `DOCUMENT.REGEXES.*` | Правила из `[app.regexes.*]` |
| `VARIABLES.*` | Объявленные переменные приложения |
| `INPUT.*` | Входы текущей функции |
| `UNSTANDARD_HEADERS.REQUEST` / `UNSTANDARD_HEADERS.RESPONSE` | Заголовки, пойманные сниффером |
| `CAPTURED_URLS.REQUEST` / `CAPTURED_URLS.RESPONSE` | URL-адреса, пойманные сниффером |
| `COOKIES` | Текущее хранилище cookies |
| `LOCAL_STORAGE` | Текущее localStorage |
| `SESSION_STORAGE` | Текущее sessionStorage |
| `FUNCRESULT.<func>.JSON` / `TEXT` / `IMAGE` | Результат другой функции внутри `examples.inputs` |

### `FUNCRESULT`

`FUNCRESULT` - специальная ссылка для тестовых примеров.
Она разрешена только внутри `[app.func.*.examples]` в значениях `inputs.<key>`.

Формат:

```msra
<FUNCRESULT.A3A417.JSON["some"]["path"][0]>
<FUNCRESULT.A3A417.TEXT>
<FUNCRESULT.A3A417.IMAGE>
```

Правила:

- `JSON` позволяет дальше обращаться к вложенным объектам и массивам;
- `TEXT` и `IMAGE` дальше не продолжаются;
- синтаксис без result-kind, например `<FUNCRESULT.headers>`, невалиден;
- вне `examples.inputs` ссылка считается ошибочной.

## Верхний уровень файла

### `[misklerreverseapi]`

Этот блок хранит версию формата файла.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `version` | Версия формата MSRA-файла | Помогает инструментам понимать совместимость файла с парсером и схемой |

### `[app]`

Этот блок описывает само приложение или API-пакет, который будет сгенерирован.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `name` | Название приложения/клиента | Используется в именах пакета, заголовках и документации |
| `authors` | Список авторов | Метаданные для package-артефакта |
| `description` | Краткое описание проекта | Документация и package metadata |
| `license` | Идентификатор лицензии | Метаданные пакета |
| `version` | Версия API/клиента | Попадает в generated package и документацию |
| `timeout_ms` | Глобальный таймаут по умолчанию | Используется как базовый лимит для операций |
| `browser` | Браузер по умолчанию | Определяет runtime-поведение warmup и browser-backed функций |

#### `authors`

`authors` - массив объектов вида:

| Ключ | Что делает |
| --- | --- |
| `name` | Имя автора |
| `email` | Email автора |

#### `browser`

Допустимые значения:

- `chromium`
- `firefox`
- `webkit`
- `camoufox`

`camoufox` нужен там, где используются `humanize` и `humanize_action`.

## Warmup

### `[app.warmup]`

Warmup - это стартовая последовательность действий, которая подготавливает runtime перед вызовом функций.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `humanize` | Включает более “человеческое” поведение браузера | Снижает риск детекта автоматизации |
| `block_images` | Блокирует загрузку изображений | Ускоряет старт и снижает трафик |
| `humanize_action` | Диапазон задержки между действиями | Делает последовательность менее “роботизированной” |
| `url` | Стартовый URL warmup | Куда перейти до начала pipeline |
| `pipeline` | Список шагов warmup | Основной сценарий подготовки |
| `headers_sniffer` | Включает сбор network headers | Нужен для `UNSTANDARD_HEADERS.*` и `CAPTURED_URLS.*` |
| `on_error_screenshot_path` | Путь к скриншоту при ошибке | Позволяет сохранить диагностический снимок |
| `timeout_ms` | Таймаут warmup | Локальный override поверх `app.timeout_ms` |

### `humanize_action`

Значение вида `{from=1000, to=3000}` задаёт диапазон задержки в миллисекундах.
Оно имеет смысл только когда `humanize=true`.

### `pipeline`

`pipeline` - это массив шагов. Каждый шаг описывает одно действие или условный блок.

Поддерживаемые действия:

- `wait_sniffer`
- `wait_element`
- `element`
- `wait_network`
- `always`

| Ключ | Что делает | Где используется |
| --- | --- | --- |
| `action` | Тип шага | Всегда |
| `state` | Состояние ожидания | Для `wait_element`, `element`, `wait_network` |
| `source` | `request` или `response` | Только для `wait_sniffer` |
| `what` | CSS-селектор или имя отслеживаемого объекта | Для `wait_sniffer`, `wait_element`, `element`, `always` |
| `then` | Следующее действие после успешного условия | Для `wait_element`, `element`, `always` |
| `if` | Условие для `always` | Только для `always` |
| `raise` | Пользовательское сообщение об ошибке | Любой шаг, где нужна диагностика |
| `timeout_ms` | Локальный таймаут шага | Для точечной настройки |
| `for_tests` | Пометка шага как test-only | Для разделения обычного и тестового warmup |

#### `wait_sniffer`

| Ключ | Что делает |
| --- | --- |
| `source` | Указывает, откуда ждать событие: `request` или `response` |
| `what` | Имя заголовка или целевого поля |

#### `wait_element` и `element`

| Ключ | Что делает |
| --- | --- |
| `state` | `visible`, `hidden`, `attached`, `detached` |
| `what` | CSS-селектор элемента |
| `then` | Что сделать после нахождения элемента |

`wait_element` ждёт условие, `element` проверяет его как более немедленную операцию.

#### `wait_network`

| `state` | Что означает |
| --- | --- |
| `load` | Страница загрузилась |
| `domcontentloaded` | DOM готов |
| `networkidle` | Сеть успокоилась |
| `commit` | Ответ получен, но рендер ещё может продолжаться |

#### `always`

`always` - это шаг, который выполняется независимо от условия, но может иметь `if` и `then`.
Он нужен для fallback-сценариев и ветвления warmup.

## Variables

### `[app.variables.<name>]`

Переменные описывают значения, которые runtime может извлекать из заголовков, storage или других источников и затем использовать как обычные reference-значения.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `types` | Список возможных типов/веток значения | Описывает, каким может быть переменная |
| `description` | Описание переменной | Документация и подсказки |
| `read_only` | Запрещает setter в generated package | Делает переменную capture-only |
| `from` | Источник значения | Откуда runtime должен взять значение |

### `types`

`types` - массив type-веток. Одна переменная может иметь несколько допустимых форм.

| Ключ | Что делает |
| --- | --- |
| `type` | Базовый тип: `string`, `integer`, `boolean`, `null`, `array`, `object` |
| `revalue` | Ограничение/валидация значения: диапазон, regex-ссылка или другой допустимый спецификатор |
| `value` | Фиксированное значение |

Правила:

- `value` и `revalue` вместе использовать нельзя;
- `revalue` нужен, когда значение нужно валидировать, а не просто зафиксировать;
- `value` используется как явный допустимый literal.

### `from`

`from` - это reference на источник значения.

Типичные источники:

- `UNSTANDARD_HEADERS.REQUEST.x-city`
- `UNSTANDARD_HEADERS.REQUEST.x-language`
- `UNSTANDARD_HEADERS.REQUEST.x-key`
- `COOKIES.*`
- `LOCAL_STORAGE.*`
- `SESSION_STORAGE.*`
- другие разрешённые reference-ветки

Практический смысл:

- `from` связывает runtime-данные с переменной;
- переменная потом доступна через `VARIABLES.<name>`;
- это не “константа”, а точка связывания с живым источником.

## Prefixes

### `[app.prefixes]`

Префиксы - это именованные константы для повторно используемых URL-частей, origin-ов, base API адресов и других строковых шаблонов.

Каждый ключ в `[app.prefixes]` становится reference-root:

```msra
[app.prefixes]
BASE_API="https://api.example.com"
ORIGIN="https://example.com/"
```

Дальше они используются как:

```msra
<DOCUMENT.PREFIXES.BASE_API>
<DOCUMENT.PREFIXES.ORIGIN>
```

Зачем это нужно:

- не дублировать одни и те же URL по всему файлу;
- централизованно менять base-адреса;
- строить URL через составные выражения.

## Regexes

### `[app.regexes.<name>]`

Regexes описывают правила валидации строк и optional preprocessing перед проверкой.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `regex` | Регулярное выражение | Основная проверка значения |
| `actions` | Предобработка строки | Нормализация перед regex |
| `raise` | Текст ошибки | Сообщение, если значение не проходит |
| `description` | Описание правила | Документация и подсказки |

### `actions`

Каждое действие описывается inline table:

| Ключ | Что делает |
| --- | --- |
| `action` | `lower`, `upper`, `capitalize`, `trim` или `replace` |
| `what` | Что искать при `replace` |
| `with` | На что заменять при `replace` |

Правила:

- `what` и `with` нужны только для `replace`;
- `what` и `with` должны быть статическими, без динамических ссылок;
- actions применяются до regex-проверки.

### Когда это полезно

- привести текст к нижнему регистру;
- заменить пробелы на `+` или другой separator;
- убрать лишние пробелы;
- унифицировать ввод перед regex.

## Groups

### `[app.groups.<path>]`

Группы - это дерево логических разделов API.
Они влияют на структуру generated package, на организацию классов и на визуальное группирование методов.

| Ключ | Что делает |
| --- | --- |
| `description` | Описание группы |

Зачем нужны группы:

- разделяют API по смыслу;
- помогают строить иерархию классов;
- делают package structure читаемой;
- улучшают навигацию в IDE и документации.

## Functions

### `[app.func.<id>]`

Это основная единица языка: декларация функции API.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `name` | Имя метода в generated package | Определяет публичное имя функции |
| `transport` | Способ выполнения | `direct`, `fetch` или `goto` |
| `method` | HTTP-метод | `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS` |
| `group` | Ссылка на группу | Определяет место функции в package tree |
| `description` | Описание функции | Документация и docstring |

### `transport`

| Значение | Что делает |
| --- | --- |
| `direct` | Прямой HTTP-клиент, без браузерного page flow |
| `fetch` | Браузерный/HTTP-fetch сценарий |
| `goto` | Переход в браузере на страницу и последующая обработка результата |

### `method`

HTTP-метод определяет способ запроса.

Для `goto` метод обычно не нужен и валидация должна его отклонять как несовместимый.

### `group`

`group` связывает функцию с `[app.groups.*]`.
Это влияет на структуру generated package и на то, как методы будут сгруппированы в клиенте.

## Function inputs

### `[app.func.<id>.input.<name>]`

Каждый input - это аргумент метода.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `type` | Тип аргумента | Формирует сигнатуру и validation |
| `description` | Описание аргумента | Подсказки и docstring |
| `required` | Обязателен ли аргумент | Влияет на сигнатуру и runtime-check |
| `default` | Значение по умолчанию | Если аргумент не передан |
| `values` | Набор допустимых значений | Ограничивает выбор и может сузить тип |
| `data` | Служебная/связующая ветка значения | Используется там, где input связан с другим источником |
| `revalue` | Ограничение диапазона или формата | Валидация значения |

### `type`

Поддерживаются:

- примитивы: `string`, `integer`, `boolean`, `null`, `array`, `object`;
- list-форма: `list[string]`, `list[integer]` и т.п.;
- объединения через массив типов: `[string, null]`.

### `default`

`default` задаёт значение, которое используется, если caller не передал аргумент.
Если input не `required`, generated signature обычно получает `None` или явный default.

### `values`

`values` ограничивает допустимые значения input-а.
Это удобно для enum-подобных параметров, например сортировки или режима.

Если одновременно есть `revalue`, то `values` обычно становится дополнительной документацией, а не способом динамически сузить тип.

### `data`

`data` - это связка input-а с источником значения.
Чаще всего это reference или другой контрактный источник, который потом используется в `url.params`, `body` или других вычисляемых местах.

В документации на практике полезно воспринимать его как “откуда берётся значение, если оно не вводится вручную”.

### `revalue`

`revalue` описывает, какие значения допустимы, когда одного типа недостаточно.

Типичные формы:

- числовой диапазон: `{from=1, to=27}`
- reference на regex-правило: `<DOCUMENT.REGEXES.COUNTRY_ALIAS>`

Зачем нужен `revalue`:

- ограничить число;
- проверить формат строки;
- привязать input к уже объявленному regex.

## URL

### `[app.func.<id>.url]`

`base` - базовая часть адреса функции.
Она может собираться из префиксов и literal-кусочков.

| Ключ | Что делает |
| --- | --- |
| `base` | Базовый URL или путь |

### `[app.func.<id>.url.params.<name>]`

`url.params` описывает query-параметры и path-like подставляемые части URL.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `sub_url` | Помечает параметр как подпуть | Удобно различать sub-path и query-параметр |
| `required` | Обязателен ли параметр | Влияет на валидацию и генерацию |
| `list` | Может ли параметр быть списком | Нужен для повторяющихся значений |
| `list_style` | Как именно сериализовать список | `repeat`, `delimited`, `bracket`, `json` |
| `values` | Допустимые/отображаемые значения | Позволяет управлять enum-логикой и URL-mapping |
| `description` | Описание параметра | Документация |
| `data` | Источник значения | Из какого input/reference брать данные |
| `revalue` | Ограничение на значение | Валидация input-а, с которым связан параметр |

#### `list_style`

| Ключ | Что делает |
| --- | --- |
| `style` | Способ сериализации списка |
| `delimiter` | Разделитель для `delimited` |
| `indexed` | Использовать ли индексы в `bracket` |

Доступные стили:

- `repeat`: `tag=a&tag=b`
- `delimited`: `tag=a,b`
- `bracket`: `tag[]=a&tag[]=b` или `tag[0]=a&tag[1]=b`
- `json`: `tag=["a","b"]`

#### `values`

Каждый item в `values`:

| Ключ | Что делает |
| --- | --- |
| `value_in_url` | Что реально попадёт в URL |
| `value` | Семантическое значение, которое выбирает caller |
| `default` | Флаг fallback-варианта |

Правила:

- `value` обязателен, если `default=true` не указан;
- `default=true` делает запись fallback-значением, но не делает её “выбираемой” сама по себе;
- `value` должен быть статическим, без dynamic references;
- если `list=true` и `data` есть, нужен хотя бы один selectable `value`, если не задан `revalue`.

## Body

### `[app.func.<id>.body]`

`body` описывает тело запроса.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `type` | MIME type тела | Определяет формат сериализации |
| `charset` | Charset или `null` | Нужен для некоторых body-форматов |
| `boundary` | Boundary для multipart | Обязателен для `multipart/form-data` |
| `return_name` | Сохранять/возвращать имя части | Полезно для multipart/file-like частей |
| `filename` | Имя файла в payload | Для файловых частей и загрузок |
| `data` | Само тело / payload | Значение, которое сериализуется |

### Основные правила body

- `multipart/form-data` требует `boundary`;
- `boundary` запрещён для non-multipart body;
- `application/x-www-form-urlencoded` требует либо `data`, либо nested `url`-ветку;
- nested body tables допустимы только внутри `multipart/form-data`;
- `url`-ветка внутри body используется для nested form-url-encoded сценариев.

### Body item tables

`body.<name>` - это вложенный body-item, обычно описывающий часть multipart payload или nested field.

Примеры:

- обычный JSON-like item: `body.ANYNAME`
- вложенный multipart item: `body.VLOJENNOST.ANYNAME2`
- urlencoded nested item: `body.VLOJENNOST.ANYNAME3.url`

Для таких таблиц важно:

- родитель должен быть `multipart/form-data`, если есть вложенность;
- `url` нельзя ставить раньше body-item;
- если имя `url` нужно буквально, его можно записать в кавычках.

### Для чего нужны `return_name` и `filename`

- `return_name=true` подчёркивает, что часть должна сохранять своё поле/имя при сериализации;
- `filename` задаёт имя файла для file-like или бинарных частей;
- оба поля особенно полезны для multipart и загрузок.

## Headers

### `[app.func.headers]`

Это общий блок заголовков и request policy для всех функций.
Он задаёт дефолты, которые потом могут быть переопределены в `[app.func.<id>.headers]`.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `referrer` | Значение referrer | Управляет referer/referrer policy |
| `cors_mode` | Режим fetch CORS | `cors`, `no-cors`, `same-origin` |
| `credentials` | Credentials policy | `omit`, `same-origin`, `include` |
| `headers` | Дополнительные заголовки | Базовый набор HTTP headers |

### `[app.func.<id>.headers]`

Локальный блок headers для конкретной функции.
Он переопределяет общие значения из `[app.func.headers]`.

Зачем нужен:

- задать особый referrer только для одной функции;
- изменить CORS/credentials policy;
- добавить или заменить заголовки для конкретного endpoint-а.

## Postprocess

### `[app.func.<id>.postprocess]`

`postprocess` применяется для `goto`-сценариев и других функций, где результат надо извлечь из страницы, а не из чистого HTTP-ответа.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `render_html` | Рендерит HTML-ответ | Нужен, если надо получить видимый DOM/HTML |
| `goto_pipeline` | Дополнительный pipeline после перехода | Позволяет добрать данные с страницы |
| `evaluate` | Путь к JS-скрипту извлечения | Для кастомной логики разбора страницы |

### `evaluate`

Это путь к JavaScript-файлу, который извлекает или преобразует данные после открытия страницы.
Обычно такой файл лежит рядом с функцией в `postprocess/`.

## Examples

### `[app.func.<id>.examples]`

Примеры описывают тестовые сценарии и контрактные фикстуры для функции.

| Ключ | Что делает | Зачем нужен |
| --- | --- | --- |
| `examples` | Массив примеров | Список входных сценариев |
| `test` | Глобальный флаг для блока | Помечает, нужен ли этот блок как тестовый контракт |

### `examples` items

Каждый элемент массива `examples` описывает один сценарий.

| Ключ | Что делает | Обязательность |
| --- | --- | --- |
| `inputs` | Значения входов для примера | Обязательно |
| `file` | Путь или имя snapshot/fixture-файла | Необязательно |
| `test` | Локальный флаг примера | Необязательно |

Пример:

```msra
[app.func.A3A417.examples]
examples=[
    {"inputs"={"query"="example"}, "test"=false, "file"="local1.json"},
    {"inputs"={"query"="example"}}
]
test=true
```

### Что важно в examples

- `inputs` обязателен, потому что без него пример не описывает сам сценарий вызова;
- `file` опционален, потому что не все примеры обязаны ссылаться на snapshot;
- `FUNCRESULT` можно использовать только внутри `inputs` значений;
- `test=true` делает блок кандидатом на контрактное тестирование.

## Как думать о настройках

Если упростить язык до логической модели, то:

- `misklerreverseapi` описывает формат файла;
- `app` описывает сам пакет;
- `warmup` описывает стартовое поведение runtime;
- `variables` описывают значения, извлекаемые из окружения;
- `prefixes` и `regexes` описывают общие повторно используемые куски;
- `groups` задают структуру клиентского дерева;
- `func` задаёт конкретные API-методы;
- `input`, `url`, `body`, `headers`, `postprocess`, `examples` описывают контракт и runtime каждой функции.

Если нужна визуальная семантика и правила сравнения путей, см. [Пути и identity](msra-paths.md).
