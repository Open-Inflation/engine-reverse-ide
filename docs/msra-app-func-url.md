# [app.func.<id>.url] и [app.func.<id>.url.params.<name>]

Таблица `[app.func.<id>.url]` задаёт базовый URL функции, а вложенные таблицы `[app.func.<id>.url.params.<name>]` описывают query-параметры и path-like части URL.

## `[app.func.<id>.url]`

| Ключ | Что делает | Как влияет на поведение |
| --- | --- | --- |
| `base` | Базовый URL или относительный путь | Становится стартовой строкой, к которой URL builder добавляет параметры и которая определяет endpoint метода. Значение может быть литералом или reference-выражением. |

### Что меняется

- `base` определяет, куда именно пойдёт запрос.
- Если `base` не задан, значение может браться из аргумента `url`, если такой input есть у функции.
- Обычно `base` указывают явно, чтобы контракт был предсказуемым и самодокументируемым.

## `[app.func.<id>.url.params.<name>]`

Каждый такой блок описывает один параметр URL.

| Ключ | Что делает | Как влияет на поведение |
| --- | --- | --- |
| `@SubUrl` | Помечает параметр как часть составного URL-пути | Используется для сегментов адреса, которые читаются как вложенный путь, а не как обычный query key. Полезна для маршрутов с подстраницами, локалями, кодами магазинов, slug-ами и другими path-like фрагментами. |
| `@List` | Делает параметр списочным | Разрешает список значений, несколько fallback-значений в `values` и отдельную сериализацию списка. |
| `list_style` | Формат сериализации списка | Управляет тем, как список попадёт в URL: `repeat`, `delimited`, `bracket` или `json`. |
| `const` | Фиксированное значение | Делает параметр неизменяемым для caller-а. |
| `values` | Набор допустимых семантических значений | Позволяет сделать mapping `value -> value_in_url` и определить fallback-значения через `default=true`. |
| `description` | Описание параметра | Используется в документации и подсказках. |
| `from` | Источник значения | Берёт значение из `INPUT`, `VARIABLES` или другого допустимого reference-источника и подставляет его в URL. |
| `match` | Ограничение формата | Проверяет значение как диапазон или regex-правило. |

### `list_style`

| Ключ | Что делает |
| --- | --- |
| `style` | Способ сериализации списка |
| `delimiter` | Разделитель для `delimited` |
| `indexed` | Использовать ли индексы при `bracket`-сериализации |

Доступные стили:

- `repeat`: `tag=a&tag=b`
- `delimited`: `tag=a,b`
- `bracket`: `tag[]=a&tag[]=b` или `tag[0]=a&tag[1]=b`
- `json`: `tag=["a","b"]`

### `@SubUrl`

`@SubUrl` помогает читать параметр как адресный сегмент, который участвует в построении пути.
Это удобно там, где URL строится не только из query string, а из комбинации базового адреса и вложенных path-like частей.

Когда такой параметр используется вместе с `values`, можно описать семантическое значение и то, как оно кодируется в URL.

### `values`

Каждый item в `values` может содержать:

| Ключ | Что делает |
| --- | --- |
| `value_in_url` | Что реально попадёт в URL |
| `value` | Семантическое значение, которое выбирает caller |
| `default` | Fallback-значение |

### Правила

- Если item помечен `default=true`, он считается fallback-значением.
- Если в `values` есть и selectable, и default entries, client использует selectable items как выбор, а default items как запасной вариант.
- Если `@List` нет, несколько `default=true` entries запрещены.
- Если `@List` есть и `from` ссылается на `INPUT.<name>`, input должен быть списком, иначе анализатор выдаст ошибку.
- Если `values` и `match` заданы вместе, это конфликт; нужно выбрать либо enum-mapping, либо формальную валидацию.
- `const` конфликтует с `values`, `from` и `match`, потому что описывает уже не выбор, а фиксированный URL-аргумент.

## Нестандартные namespace-ветки

`url.params` может содержать вложенные `params` namespaces.

Это позволяет описывать составные sub-URL конструкции, но у таких веток есть строгие правила пути:

- после параметра можно открывать только один вложенный `params`-namespace;
- если нужен literal-элемент с именем `params`, его нужно экранировать кавычками;
- `@SubUrl` не отменяет общие правила path identity, см. [Пути и identity](msra-paths.md).
