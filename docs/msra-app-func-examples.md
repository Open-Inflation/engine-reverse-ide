# [app.func.<id>.examples]

Таблица `[app.func.<id>.examples]` хранит именованные примеры вызова функции. Это namespace-контейнер: сам блок может быть пустым, а реальные примеры живут в `[app.func.<id>.examples.<name>]`.

## `[app.func.<id>.examples.<name>]`

| Ключ | Что делает | Как влияет на поведение |
| --- | --- | --- |
| `inputs` | Значения входов для примера | Опциональное поле. Если его не указать, оно считается равным `inputs={}` и пример всё равно остаётся валидным. |
| `type` | Формат результата примера | Опциональное поле. Значения: `json` по умолчанию, `text` для текстовых ответов и `image` для image-стримов. |
| `@Test` | Помечает пример как тестовый контракт | Делает пример видимым как проверяемый сценарий API-контракта. |
| `@Docs` | Помечает пример как документируемый сценарий | Делает пример пригодным для показа в документации и как пользовательский sample. |
| `description` | Краткое описание примера | Используется генератором для README и mini-docs, поэтому лучше держать его коротким и статическим. |

## `inputs`

`inputs` должен быть inline table с ключами, которые совпадают с именами объявленных input-ов функции. Если `inputs` не указан, анализатор трактует пример как `inputs={}`.

### Что меняется

- анализатор проверяет, что каждый `inputs.<key>` действительно существует в функции;
- анализатор проверяет типы и допустимые значения;
- `FUNCRESULT` можно использовать только внутри `inputs.<key>.value`, чтобы связать один пример с результатом другой функции; ссылка должна указывать имя source example перед result-kind, например `<FUNCRESULT.<function>.<example>.JSON[0]["id"]>`;
- если текущий пример помечен `@Docs` или `@Test`, то referenced example с `FUNCRESULT` должен иметь те же флаги; если у текущего примера флага нет, проверку для него не выполняем;
- `type` управляет тем, как генератор покажет пример в README: `json` по умолчанию, `text` для строкового ответа, `image` для image-stream;
- `description` можно использовать как короткую статическую подпись для документации и generated README;
- generated README/example.py/quick_start.rst включают все примеры, помеченные `@Docs`; `@Test` остаётся независимым флагом для тестового контракта;
- если `@Docs` пример ссылается на не-`@Docs` пример через `FUNCRESULT`, LSP сообщает об этом, а generated README/example.py не подменяет зависимость и вместо этого падает `RuntimeError` в месте обращения;
- если `inputs` отсутствует, это эквивалентно `inputs={}` и не считается ошибкой.
- имя примера становится именем переменной в generated README/example.py/quick_start.rst; если два выбранных примера сводятся к одному имени переменной, LSP покажет ошибку до генерации.

## `@Test` и `@Docs`

### Семантика

- `@Test` помечает пример для contract testing.
- `@Docs` помечает пример для отображения в документации.

## Правила

- `FUNCRESULT` допускается только внутри example inputs и должен указывать source example по имени, например `<FUNCRESULT.<function>.<example>.JSON>`.
- Если пример ссылается на несуществующий input или даёт значение неподходящего типа, analyzer выдаёт диагностику.
