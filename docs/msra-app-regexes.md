# [app.regexes.<name>]

Таблица `[app.regexes.<name>]` описывает повторно используемое правило валидации строки. Это именованный regex-конракт, который можно ссылаться из других частей MSRA.

| Ключ | Что делает | Как влияет на поведение |
| --- | --- | --- |
| `regex` | Основное регулярное выражение | Определяет, какое значение считается валидным после предобработки. |
| `actions` | Цепочка нормализаций перед проверкой | Приводит строку к нужному виду до сравнения с `regex`. Поддерживаются `lower`, `upper`, `capitalize`, `trim` и `replace`. |
| `raise` | Пользовательское сообщение об ошибке | Показывается, если значение не проходит проверку. |
| `description` | Описание правила | Используется в документации, подсказках и hover-информации. |

## `actions`

Каждое действие задаётся inline table.

| Ключ | Что делает |
| --- | --- |
| `action` | Тип нормализации: `lower`, `upper`, `capitalize`, `trim` или `replace` |
| `what` | Что искать при `replace` |
| `with` | На что заменять при `replace` |

### Правила

- `what` и `with` нужны только для `replace`.
- `what` и `with` должны быть статическими.
- `actions` применяются до regex-проверки.

## Что меняется

- `regexes` позволяют переиспользовать одно и то же правило в нескольких input-ах, URL-параметрах или variable-match-ах.
- Если значение не проходит `regex`, runtime использует `raise`, если оно задано.
- Нормализация из `actions` учитывается на уровне анализа контракта и tooling.
